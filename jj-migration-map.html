jj Migration Map

Holistic plan for moving the nixfiles git tooling to Jujutsu — colocated, incremental, reversible. Worktree-first workflow retained.

Core strategy: nothing is big-bang. Run jj colocated (jj git init --colocate) so every repo stays a normal git repo and git remains a permanent escape hatch. Add a programs.jujutsu nix module alongside the existing git config rather than replacing it; retire git-specific bits only once each is confirmed unused. Adopt per-repo, starting with nixfiles itself.

Chosen workflow — keep worktrees (Model B). Worktree use here is concurrency-driven (parallel work, holding a branch's state open during review) and doubles as an in-flight TODO list — not a workaround for painful switching. So worktrees stay: c / w / bd / cl are retained and re-pointed at jj workspaces, with auto-update-stale enabled so restacks reconcile siblings transparently. Drifting toward in-place editing (Model A) is allowed to happen naturally, but is not a goal.

Workstream 2 — worktree tooling design

One set of commands; conversion is explicit and whole-repo. c/w/bd transparently dispatch on whether the repo is jj (test -d <root>/.jj/repo): jj impl if converted, git impl otherwise — no g-prefixed command to remember (the git impl is the fallback the dispatcher calls; keep gc/gw/gbd only as an explicit "force git" escape hatch). A repo is converted in one shot by jj-init, never half-and-half. Working dirs become jj workspaces (non-colocated — no git CLI inside; the bare store is shared).

ToolBehaviorNotes
jj-init
eager full conversion		 Converts a whole existing git repo to jj (validated). Locate the bare via git rev-parse --git-common-dir → set the interop refspec (+refs/heads/*:refs/remotes/origin/*) → jj git init --no-colocate --git-repo=.git <root> → jj git fetch + track bookmarks → jj workspace forget default (bare .jj at root). Then enumerate git worktree list --porcelain and for every worktree incl. main/master: assert clean → git worktree remove → jj workspace add --name <n> -r <branch>. All-or-nothing: the clean check is a pre-flight before jj git init — if any worktree is dirty, abort listing them and touch nothing (a store with no converted workspaces is itself the broken half-state, and would flip c/w into jj-mode over still-git worktrees). Recovery: commit/stash the listed trees, re-run. End state: every dir is a jj workspace, only the bare git worktree remains. For existing git repos only (keeps the bare .git as a transitional fallback). New clones skip this: clone uses native jj git clone --no-colocate → no .git at all, fully jj-native. clone -g/--git (used by gw) opts back into plain git.
c / w / bd [-r]
auto-dispatch		 If the repo is jj: jj workspace add off current @ (fallback trunk()) + bookmark-per-branch + claude-symlinks + cd; bd [-r] = jj workspace forget + rm + bookmark delete (-r first advances the default bookmark). If the repo is plain git: today's git worktree behavior. Existing dir → just cd. rbd folds into bd -r. Same short names, now jj-or-git automatically.
gc / gw / gbd [-r]
git impl / escape hatch		 The git implementation the dispatcher falls back to; also callable directly to force git worktrees even in a jj repo. No longer the primary fallback mechanism — just an override.

Store-anchor — resolved (Option D, verified): the earlier worry was that .jj/repo lives in the first workspace, so deleting that dir destroys the store. We avoid it entirely: jj git init <root> puts the store at <repo>/.jj (root), then jj workspace forget default drops the root's own working copy. The store now sits at the root — the direct analogue of bare .git — owned by no workspace, so every real workspace (incl. main) is freely deletable and the root never snapshots stray files. No .jj-store/ anchor dir needed.

Decisions — all resolved:

  • Anchor strategy ✓ — Option D (bare .jj at root, default workspace forgotten). Strictly better than the earlier .jj-store/ sibling: one fewer dir, zero snapshot churn, exact bare-.git analogue.
  • Repo-type dispatch ✓ — c/w/bd branch on test -d <root>/.jj/repo; conversion is explicit via jj-init, never lazy in c.
  • Default base ✓ — current workspace's @ (stacking-friendly, matches today's c), falling back to trunk() when @ can't resolve (e.g. called from a git worktree / first init).
  • Bookmark on create ✓ — yes, a bookmark named after the branch (slashes preserved) at the new @, so PRs can be pushed. Skipped for the default branch. (Bookmarks don't auto-advance onto a later jj new.)
  • workspace add -r semantics ✓ — new workspace's @ = an empty change on top of <base> (verified).
  • bd -r ✓ — advances the default bookmark to the workspace's change (jj bookmark set <default> -r <change> --allow-backwards) before teardown. No auto-push.
